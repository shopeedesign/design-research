# Output Format Reference

最终回答默认用中文，除非用户要求其他语言。普通 UI 资源检索直接在聊天里给可点击 Markdown 链接；只有用户明确要求 HTML、聚合页、文件或报告页面时，才生成本地文件。

如果用户明确要求本地 HTML/聚合页，并且外链需要从静态页跳到系统浏览器：启动 `scripts/launch-external-link-helper.mjs`，HTML 外链优先通过 helper 打开；helper 不可用时，提供持久备用面板，包含完整 URL、只读输入框和复制按钮。普通聊天回答不要使用 helper 或本地中转链接。

## 回答结构

按顺序输出：

1. 研究范围：市场、平台、素材偏好和默认假设。
2. 竞品选择逻辑：为什么选这些产品。
3. 主结果表：按产品聚合资源。
4. Google 图片补充：用列表输出 Google Images 下钻核验后的来源页。
5. 非直接竞品/主题合集补充：只在有相关合集时输出。
6. 覆盖不足：说明渠道不足、未命中或受限原因。
7. 待核验线索：只放可能相关但当前未能确认的候选。

结果很多时，优先精简主结果，只保留 high fit / medium fit 的强链接。

花瓣画板默认返回站内搜索入口，不筛选具体画板；入口必须标明 `站内搜索入口 / 需浏览器核验`。

不要输出独立的“花瓣检索记录”或“集合平台检索记录”板块。花瓣入口如果保留，只放入主结果表的 `界面/截图` 列或 `待核验线索`。

## 主结果表

必须使用四列：

| 产品 | 应用商店 | 界面/截图 | 其他来源 |
|---|---|---|---|

列规则：

- `产品`：竞品应用名称，一个产品一行；不要放泛行业词、主题合集或素材平台名。
- `应用商店`：只放应用商店链接，如小米、应用宝、Google Play、ONE Store、App Store。App Store 仅在用户明确需要 iOS 时返回。
- `界面/截图`：放直接相关的截图、视频、录屏、截图合集、UI Notes、Mobbin、Chamjo、Page Flows、UI Sources、花瓣站内搜索入口、B站、YouTube、抖音等。Pinterest 不作为默认渠道，除非用户明确要求或提供链接。
- `其他来源`：放官网、帮助中心、教程文章、产品说明、社区帖等非主截图资源。

每个单元格最多保留 3-5 条最强链接，用 `<br>` 分隔。

## 链接写法

每条链接都用这个格式：

```markdown
[来源名称](https://example.com)｜素材类型｜核验状态/fit｜简短备注
```

要求：

- 聊天回答里的链接必须直达外部 `http://` 或 `https://` 页面，不使用本地跳转页、iframe、代理页或本地 HTML 中转。
- 不裸露 URL；不要把多个资源塞进一个模糊链接。
- 备注说明看到了什么页面、控件、流程、状态或局限性。

示例：

```markdown
| 闲鱼 | [小米应用商店](https://app.mi.com/...)｜应用商店截图｜已核验｜5 张当前版本截图 | [UI Notes](https://uinotes.com/...)｜截图合集｜high fit｜旧版真实截图<br>[B站教程](https://www.bilibili.com/...)｜操作视频｜仅标题/摘要支撑｜发布商品流程 | [帮助中心](https://example.com)｜官方说明｜已核验｜交易规则 |
```

## Google 图片补充

主结果表后必须增加一个列表式补充区，用 Google Images / 图片搜索发现主表外的 UI 线索。先根据图片结果判断是否有真实 UI 信息，再打开来源页二次核验；最终只返回来源页链接。

输出标题固定为 `Google 图片补充`。如果有符合结果，输出 5-8 条；不足 5 条时只输出已核验的合格结果，不用低价值链接凑数。每条使用列表样式：

```markdown
- [来源标题](https://example.com)｜来源类型｜fit/核验状态｜图片结果可见 UI + 来源页二次判断结论
```

如果没有符合结果，输出：

```markdown
Google 图片补充：没有符合结果。
```

Google 图片补充的链接规则：

- 不能返回图片直链、Google Images 搜索页、`imgurl`、`gstatic`、`encrypted-tbn`、CDN 图片文件或只打开单张图片的页面。
- 可以返回文章、视频、截图合集、教程页、应用市场页、设计案例页或含真实截图的网页。
- 图片结果只作为初筛证据；来源页打不开、来源页无 UI、来源页与图片不一致、只有 Logo/海报/宣传图/人物/插画时，不进入补充列表。
- 备注要说明图片结果看到了什么 UI，以及来源页下钻后为什么仍然可用或有什么局限。

## Fit 标签

- `high fit`：清晰展示目标功能、页面、状态或交互。
- `medium fit`：展示真实 UI 或相关流程，但没有直接展示精确目标。
- `low fit`：弱相关；默认不放主结果。
- `do not return`：纯宣传、无可见 UI、无真实使用过程、无法核验或不相关。

宁可少，也不要用低价值链接凑数。

花瓣站内搜索入口不要标 `high fit` 或 `medium fit`。使用 `站内搜索入口 / 需浏览器核验`，并说明当前自动环境无法确认画板数量、排序或内部内容。

## 地区口径

- 中国大陆：`应用商店` 优先小米应用商店、应用宝；`界面/截图` 优先 UI Notes、花瓣、B站、抖音、应用市场可见截图。
- 全球/欧美：`应用商店` 用 Google Play，App Store 仅用户明确需要 iOS 时使用；`界面/截图` 用 Mobbin、Page Flows、UI Sources、YouTube、官网/帮助中心。
- 日本：Google Play JP、YouTube 日文教程、Appliv、官网/帮助中心；App Store JP 仅用户明确需要 iOS 时使用。
- 韩国：Google Play KR、ONE Store、YouTube 韩文视频、Naver Blog、官网/帮助中心；App Store KR 仅用户明确需要 iOS 时使用。
- 东南亚/亚洲：Chamjo、Mobbin、UI Sources、Google Play 本地区、YouTube、TikTok、Page Flows、官方帮助中心。

Chamjo、Mobbin、UI Sources、Page Flows、UIguana 都属于 `界面/截图` 列。

## 主题合集

行业 UI 画板、App 截图合集、App flow collection 等主题素材不要混入主表。放到 `非直接竞品/主题合集补充`，并说明它不是直接竞品，只能参考结构、控件、状态或风格。Pinterest 只在用户明确要求、提供链接或需要灵感补充时出现。

## 覆盖不足

必须简短说明：

- 哪些竞品没有 high fit 资源。
- 哪些渠道未命中可核验结果。
- 是否存在登录、地区、索引、截图过旧、视频不可见或自动抓取受限。
- 社媒/社区平台只能写 `公开搜索未找到可核验结果`，不要写绝对没有。

## 待核验线索

只放可能相关但当前未能确认的候选；每条说明原因，如页面无法访问、需要登录、地区受限、只有标题摘要或视频内容不可见。不要给待核验线索标 high fit / medium fit。

花瓣站内搜索入口或用户提供的花瓣链接可进入待核验线索。推荐写法：

```markdown
[产品名 app - 花瓣画板搜索](https://huaban.com/search?type=board&q=...)｜站内搜索入口｜需浏览器核验｜当前自动环境无法确认画板数量、排序或内部内容
```

## 最终自检

- 结果按产品聚合，而不是按素材打散。
- 应用商店、界面/截图、其他来源不混列。
- 每条链接有来源、素材类型、核验状态或 fit、参考价值。
- Google 图片补充以列表输出 5-8 条来源页链接；如果没有合格来源页，明确写“没有符合结果”。
- Google 图片补充不返回图片直链或 Google Images 搜索页，且每条都经过来源页二次判断。
- 不编造链接、标题、页面内容、截图或视频细节。
- 花瓣、Chamjo、Mobbin 等集合型结果说明是否需要逐条核验。
- 花瓣是否只返回站内搜索入口，而不是使用 `site:huaban.com` 外部搜索结果。
- Pinterest 是否未作为默认/P0 渠道出现。
