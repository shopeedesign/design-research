# Output Format Reference

最终回答默认用中文，除非用户要求其他语言。普通 UI 资源检索直接在聊天里给可点击 Markdown 链接；只有用户明确要求 HTML、聚合页、文件或报告页面时，才生成本地文件。

本文件只规定最终结果怎么呈现。目标平台选择、固定查询词、Google Images / Google Videos、花瓣、应用商店去重等检索规则，以 `references/platform-search.md` 为准。

## 输出原则

- 资源结果以 `直接竞品主结果` 为核心；`补充参考` 只在未进入用户选择/指定流程时输出。
- 主结果按直接竞品聚合，不按素材来源打散。
- 补充参考不能混进主竞品表伪装成直接竞品。
- 如果用户未指定具体产品，第一轮只输出候选竞品编号清单，让用户最多选择 2 个；不要在同一轮输出 UI 资源。
- 用户通过编号选择或直接指定产品后，只返回用户选择/指定产品的资源，最多 2 个；不要补充非用户指定的资源。
- 未进入用户选择/指定流程、且任务明确要求直接汇总行业资源时，主结果才返回 4 个直接相关产品，补充参考返回 2 条资源。
- 主结果只保留 `high fit` 或 `medium fit` 的资源；宁可少，不用低价值链接凑数。
- 每个直接竞品在 `应用商店` 列必须只返回 1 条非 iOS 应用商店截图来源；禁止调用或返回 iOS App Store / Apple App Store。
- Google Images / Google Videos 只是发现入口，最终只返回下钻后的真实来源页；不要输出 Google 搜索结果页、图片/视频直链、缩略图链接或 CDN 资源。
- 花瓣只返回站内搜索入口或用户提供链接，不判断站内结果数量、排序或具体画板内容。
- 聊天回答里的链接必须直达外部 `http://` 或 `https://` 页面，不使用本地跳转页、iframe、代理页或本地 HTML 中转。

## 标准回答结构

如果用户未指定具体产品，先输出候选竞品选择清单，并停在这里等待用户回复：

```markdown
我先列出直接相关竞品，请选择最多 2 个序号，我再继续搜索 UI 资源：
1. 产品A
2. 产品B
3. 产品C
4. 产品D
5. 产品E
6. 产品F
```

候选选择清单不要包含应用商店、截图、视频、补充参考、覆盖不足或待核验线索。

如果用户已指定 1-2 个产品，或已经从候选清单中选择 1-2 个序号，再按顺序输出：

1. `研究范围`
   - 说明目标市场、平台范围、素材偏好和默认假设。
   - 如果用户指定功能、页面或状态，说明本次会在候选素材池中下钻判断该目标。

2. `检索入口与竞品选择`
   - 说明本次属于直接竞品、产品类型/市场、模糊功能或混合输入。
   - 说明直接竞品主结果的选择逻辑。

3. `主结果表`
   - 用固定四列表按直接竞品聚合资源。
   - 用户已选择或指定产品时，只返回这些产品，最多 2 个；不要补充其他竞品。
   - 未进入用户选择/指定流程、且任务明确要求直接汇总行业资源时，才返回 4 个直接相关产品；若候选超过 4 个，按同类程度、目标匹配度、可核验度和界面信息量筛选前 4 个。
   - 每个直接竞品最终最多返回 6 条高价值资源。
   - 每个直接竞品至少尝试覆盖两类资源；如果只能返回应用商店单链，必须在覆盖不足中说明已完成对应市场必跑渠道但没有合格来源。

4. `补充参考`
   - 用户已选择或指定产品时，省略补充参考，不输出非用户指定资源。
   - 未进入用户选择/指定流程时，只输出 2 条交互相似、素材更丰富的补充资源。
   - 可从相邻产品、非直接竞品素材、行业主题合集、App 合集或 UI 画板/board 中择优选择。
   - 每条都要说明它不是直接竞品，以及可参考的界面、流程或状态。

5. `覆盖不足`
   - 说明哪些竞品、目标平台或页面/功能缺少 high fit 资源。
   - 说明限制原因，如登录、地区、索引不足、截图过旧、视频不可见、自动抓取受限等。

6. `待核验线索`
   - 只放可能相关但当前未能确认的候选。
   - 没有待核验线索时可以省略。

## 主结果表

必须使用四列：

```markdown
| 产品 | 应用商店 | 界面/截图 | 其他来源 |
|---|---|---|---|
```

列规则：

- `产品`：直接竞品名称，一个产品一行；不要放泛行业词、主题合集、相邻产品或素材平台名。
- `应用商店`：只放 1 条非 iOS 应用商店截图链接。选择截图数量、截图质量、版本/更新时间、目标市场匹配度和访问稳定性综合最高的来源。禁止返回多个应用商店链接；禁止返回 iOS App Store / Apple App Store。
- `界面/截图`：放直接展示界面或交互的资源，如截图合集、截图/流程库、应用市场截图、视频、录屏、UI Notes、Mobbin、Chamjo、Page Flows、UI Sources、B站、YouTube、抖音、TikTok、Google Images / Google Videos 下钻后的真实来源页、花瓣站内搜索入口等。
- `其他来源`：放官网、帮助中心、教程文章、产品说明、社区资料等非主截图资源。若来源内含真实 UI 截图或操作路径，备注里说明。

数量控制：

- 用户已选择或指定产品时，主结果表必须只包含用户选择/指定的产品，最多 2 个；不要添加其他直接竞品或相邻产品。
- 未进入用户选择/指定流程、且任务明确要求直接汇总行业资源时，主结果表才包含 4 个直接相关产品；候选不足 4 个时，只返回可核验的直接相关产品，并在 `覆盖不足` 说明不足原因，不用弱相关产品凑数。
- 每个直接竞品最终最多返回 6 条高价值资源。
- `应用商店` 必须且只能返回 1 条非 iOS 来源；如果没有可核验的非 iOS 应用商店截图，写 `未找到可核验的非 iOS 应用商店截图`，不要用 App Store 替代。
- `界面/截图` 优先保留能看到真实 UI、流程或状态的资源。
- `其他来源` 保留能补充页面、流程、状态或使用方式的辅助资源。
- 同一产品下必须去重；相同来源页通过多个查询词命中时只保留一次。

最低覆盖：

- 每个进入主结果表的直接竞品，至少尝试覆盖 `应用商店`、`界面/截图`、`视频/官方/教程` 中的两类资源；应用商店仅限 1 条非 iOS 来源。
- 如果应用商店之外有 high fit / medium fit 来源，不能只返回应用商店链接。
- 如果最终只能返回应用商店单链，必须在 `覆盖不足` 中写明已检索目标市场必跑渠道、必要兜底渠道和名称变体，但没有找到其他可核验来源。
- 产品类型/市场检索下，直接竞品的最低覆盖优先于补充参考。

同一单元格多条链接用 `<br>` 分隔。

## 链接格式

每条链接使用统一格式：

```markdown
[来源名称](https://example.com)｜素材类型｜核验状态/fit｜简短备注
```

写法要求：

- `来源名称`：写真实来源名称，如 Google Play、AppBrain、Mobbin、YouTube、官方帮助中心；不要把 Google Images / Google Videos 写成最终来源。
- `素材类型`：使用明确类型，如 `应用商店截图`、`截图合集`、`流程库`、`操作视频`、`官方帮助截图`、`教程文章`、`站内搜索入口`。
- `核验状态/fit`：使用 `high fit`、`medium fit`、`已核验`、`仅标题/摘要支撑`、`缩略图可见`、`画面未完整核验`、`站内搜索入口 / 需浏览器核验` 等。
- `简短备注`：说明看到的页面、流程、控件、状态或局限；不要写空泛评价。

示例：

```markdown
| JobStreet | [Google Play](https://play.google.com/...)｜应用商店截图｜high fit｜岗位搜索、推荐、一键申请、申请追踪截图 | [YouTube](https://www.youtube.com/...)｜操作视频｜画面未完整核验｜JobStreet App Guide，缩略图和章节显示资料更新、申请流程<br>[AppBrain](https://www.appbrain.com/...)｜截图页｜medium fit｜多张安卓截图备份 | [帮助中心](https://example.com)｜官方说明｜medium fit｜申请流程说明，截图有限 |
```

## Fit 标签

- `high fit`：清晰展示用户目标功能、页面、状态或交互。
- `medium fit`：展示真实 UI 或相关流程，但未直接覆盖精确目标。
- `low fit`：弱相关；默认不放主结果。
- `do not return`：纯宣传、无可见 UI、无真实使用过程、无法核验或不相关。

特殊口径：

- 花瓣站内搜索入口不要标 `high fit` 或 `medium fit`，统一写 `站内搜索入口 / 需浏览器核验`。
- iOS App Store / Apple App Store 不作为来源，不进入主结果、补充参考或待核验线索。
- 视频未完整播放时，必须说明 `画面未完整核验`、`缩略图可见` 或 `仅标题/摘要支撑`。
- B站/视频平台中标题不含 `APP` / `UI` 但可见完整软件录屏、手机界面或连续任务流程的资源，可以标 `high fit`；不要因播放量低或标题像生活教程而降级。
- Google Images / Google Videos 命中的结果必须下钻到来源页；最终按真实来源页类型归类。
- 社媒、社区、论坛类平台只能写 `公开搜索未找到可核验结果`，不要写绝对没有。

## 补充参考

补充参考必须放在独立区块，不能进入主结果表。用户已选择或指定产品时，省略补充参考，不输出非用户指定资源。未进入用户选择/指定流程时，最终只返回 2 条补充资源。

补充参考包括：

- 相邻产品：不是直接竞品，但交互流程相似、公开 UI 素材更丰富的产品。
- 非直接竞品素材：不属于同类产品，但包含相似页面、流程、状态或组件的素材。
- 行业主题合集：围绕目标行业、功能或页面整理的截图合集、案例合集、灵感合集。
- App 合集：包含多个相关 App 的榜单、评测、应用合集或截图集合。
- UI 画板/board：花瓣、Pinterest 或其他画板类资源；按 `references/platform-search.md` 的目标平台规则处理。

推荐格式：

```markdown
**补充参考**
- [来源名称](https://example.com)｜补充参考类型｜核验状态/fit｜不是直接竞品；可参考的界面、流程或状态
```

数量控制：

- 未进入用户选择/指定流程时，必须只输出 2 条补充资源。
- 优先选择交互相似、素材更丰富的相邻产品。
- 主题合集、App 合集、UI 画板只在比相邻产品更有参考价值时进入这 2 条。
- 不为凑数返回低价值链接；如果只有 1 条合格补充资源，说明另一条覆盖不足。

## 覆盖不足

覆盖不足必须具体，不要笼统写“没找到”。

应说明：

- 哪些直接竞品没有 high fit 资源。
- 哪些目标平台未命中可核验结果。
- 哪些页面、功能或状态没有被充分覆盖。
- 是否存在登录、地区、索引、截图过旧、视频不可见或自动抓取受限。
- 写“未找到/覆盖不足”前，是否已考虑 P1/P2、名称变体、地区版本、包名或开发者名等兜底。

推荐写法：

```markdown
**覆盖不足**
公开索引里没有找到 X / Y 在目标页面上的 high fit 资源；当前可核验证据主要来自应用商店、公开视频和官方帮助中心。部分视频仅能依据缩略图、标题或摘要判断，具体操作画面仍需人工播放核验。
```

## 待核验线索

待核验线索只放可能有用但当前无法确认的候选。

适合放入待核验线索的情况：

- 页面需要登录或地区访问。
- 视频无法确认具体画面。
- 搜索结果标题/摘要相关，但页面无法打开。
- 花瓣站内搜索入口或用户提供但当前打不开的画板链接。

格式：

```markdown
**待核验线索**
- [来源名称](https://example.com)｜候选线索｜需浏览器核验｜需要登录 / 地区受限 / 只有标题摘要 / 视频内容不可见
```

待核验线索不要标 `high fit` 或 `medium fit`。

花瓣入口推荐写法：

```markdown
[产品名 app - 花瓣画板搜索](https://huaban.com/search?type=board&q=...)｜站内搜索入口｜需浏览器核验｜当前自动环境无法确认画板数量、排序或内部内容
```

## HTML / 聚合页输出

只有用户明确要求本地 HTML、聚合页、文件或报告页面时，才生成本地文件。

如果外链需要从静态页跳到系统浏览器：

1. 启动 `scripts/launch-external-link-helper.mjs`。
2. HTML 外链优先通过 helper 打开。
3. helper 不可用时，提供持久备用面板，包含完整 URL、只读输入框和复制按钮。

普通聊天回答不要使用 helper 或本地中转链接。

## 最终自检

输出前检查：

- 是否说明研究范围、检索入口、竞品选择逻辑和素材偏好。
- 是否按用户选择/指定状态正确决定只输出主结果，或输出主结果加补充参考。
- 主结果是否按直接竞品聚合，而不是按素材打散。
- 如果用户未指定具体产品，是否第一轮只输出候选竞品编号清单并等待选择。
- 候选竞品清单是否要求用户最多选择 2 个序号，且没有同时输出资源。
- 如果用户已选择或指定产品，主结果是否只返回用户选择/指定的产品，最多 2 个。
- 用户选择或指定产品后，是否没有输出补充参考或其他非用户指定资源。
- 未进入用户选择/指定流程时，主结果是否只返回 4 个直接相关产品，补充参考是否只返回 2 条资源。
- 应用商店、界面/截图、其他来源是否分列正确。
- 每个直接竞品是否最多返回 6 条高价值资源。
- 每个直接竞品是否至少尝试覆盖两类资源；只返回应用商店单链时，是否写清覆盖不足。
- 同一竞品是否只返回 1 条综合价值最高的非 iOS 应用商店截图链接。
- 是否没有调用或返回 iOS App Store / Apple App Store。
- 每条链接是否有来源、素材类型、核验状态或 fit、参考价值。
- 主结果是否只保留 `high fit` / `medium fit`。
- Google Images / Google Videos 命中的结果是否都下钻到真实来源页，且没有返回搜索页、图片/视频直链、缩略图链接或 CDN 资源。
- 花瓣是否只返回站内搜索入口或用户提供链接，并标记需浏览器核验。
- 补充参考是否放在独立区块，并说明不是直接竞品、可参考什么。
- 覆盖不足是否写清限制原因，而不是绝对否定。
