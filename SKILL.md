# design-research

用于研究竞品如何处理某个具体产品功能、页面、状态或流程，并收集能直接查看真实 UI 或交互行为的素材：截图、录屏、应用市场截图、截图/流程库、画板/board、帮助中心、教程和带可见界面的官方资料。

不要扩展成泛市场研究、商业策略分析或空泛竞品分析。

## 何时加载 references

- 开始实际检索、选择或核验任何渠道、或准备写“未找到/覆盖不足”时，读取 `references/platform-search.md`。
- 准备最终回答、四列表、fit 标签、覆盖不足或待核验线索时，读取 `references/output-format.md`。
- 只讨论方法、复盘或修改 skill 时，不需要读取平台细则。

## 执行前确认

搜索前先确定：

1. 用户要找的对象：产品类别、目标功能/页面/状态、平台范围和目标市场。
2. 竞品池：用户未指定具体产品时，先推断 6-10 个直接竞品候选，只输出编号清单并让用户选择，最多选择 2 个；用户选择后再按现有检索逻辑输出所选产品资源。
3. 名称变体：具体产品要识别别名、旧名/新名、中文名/英文名、包名、开发者名和地区名。
4. 补充关键词：提取用户原始关键词、产品名、页面/功能词、平台词和行业词，供目标平台检索和下钻核验使用。
5. 渠道策略：按 `references/platform-search.md` 选择目标平台；先建立 1 个非 iOS 应用商店截图基线，再完成对应市场必跑渠道，最后根据召回和覆盖缺口考虑其他渠道兜底。


## 核心流程

### 硬性规则

- 如果用户只给出品类、市场、功能或泛行业词，未指定具体产品名，先输出候选竞品编号清单并等待用户选择；不要在这一轮搜索或输出 UI 资源。
- 候选竞品清单只列直接相关产品名称，要求用户按序号选择，最多选择 2 个。用户选择后，只检索并输出所选产品的资源，不补充用户未指定或未选择的产品资源。
- 如果用户已经明确指定 1-2 个具体产品，跳过候选选择步骤，直接按现有检索逻辑输出这些产品的资源。
- 如果用户一次指定超过 2 个具体产品，先让用户缩小到最多 2 个，再继续检索资源。
- 选定直接竞品后，先为每个直接竞品选择且只选择 1 个非 iOS 应用商店截图来源；如果多个 Android/Google Play/第三方应用市场都可用，只保留截图信息量最高的一条。
- 禁止调用、搜索或返回 iOS App Store / Apple App Store 链接，即使用户提到 iOS、iPhone、iPad 或苹果端界面；如用户明确要求苹果端，说明当前规则不使用 App Store，并改用截图库、视频、官网帮助、Google Images / Google Videos 下钻来源补足。
- 应用商店来源只负责提供基础产品截图，不替代截图库、视频和 Google Images / Google Videos 的必跑检索。
- 市场必跑渠道不可被已有应用商店结果抵消：国内必须跑 UI Notes、Bilibili、Google Images、Google Videos；全球/欧美必须跑 Mobbin、Page Flows、YouTube、Google Images、Google Videos；东南亚/亚洲必须跑 Chamjo、Mobbin、UI Sources、YouTube、Google Images、Google Videos。
- 必跑渠道完成后，再根据召回情况选择其他渠道兜底；不要为了凑数量返回低价值或无法核验的链接。
- 默认最终主结果只返回 4 个直接相关产品、补充参考只返回 2 条资源；但当用户通过编号选择或直接指定产品时，以用户选择/指定为准，只输出这些产品资源，不输出补充参考。

1. **判断检索入口**

先判断用户给出的检索入口，并选择对应策略：

	- 直接竞品：只要用户指定了具体产品名，就围绕该产品精准搜索；功能、页面、状态或地区词作为下钻关键词，并补充别名、旧名/新名、包名、开发者和地区版本。
	- 产品类型/市场：用户指定了品类、行业或地区市场但未指定具体产品时，先推断相关直接竞品，只输出编号清单并让用户最多选择 2 个；不要在同一轮输出资源。用户选择后，再逐个按产品名执行信息检索策略。
	- 模糊功能：用户只指定功能、页面或状态，且未指定具体产品、品类或市场时，先判断哪些产品或品类可能存在该功能，形成候选竞品池，只输出编号清单并让用户最多选择 2 个；用户选择后再按竞品名精准搜索并下钻该功能。
	- 混合输入：如果用户同时给出产品类型/市场和功能，以产品类型/市场为入口，功能词作为下钻筛选；如果同时给出具体产品和功能，以具体产品为入口。
	- 仍无法形成产品、品类、市场或功能关键词时，才向用户追问。

	候选清单轮的输出只包含：

	```markdown
	我先列出直接相关竞品，请选择最多 2 个序号，我再继续搜索 UI 资源：
	1. 产品A
	2. 产品B
	3. 产品C
	...
	```

	候选清单轮不要输出应用商店、截图、视频、补充参考、覆盖不足或待核验线索。

2. **分层选择参考对象**

	资源结果按以下层级处理：

	- 直接竞品：同类产品，作为主结果表的核心对象；如果用户选择或指定了产品，只保留用户选择/指定的产品，最多 2 个。
	- 补充参考：只有在用户没有通过编号选择或直接指定产品、且当前任务按原有 4+2 汇总输出时，才提供 2 条补充资源；用户选择/指定产品后不要补充非用户指定的资源。

	补充参考只能放在独立补充区，不能混进主竞品表伪装成直接竞品。用户选择或指定产品时跳过补充参考；未进入选择/指定流程时，先保证直接竞品达到最低覆盖要求，再整理补充参考。

3. **按竞品和目标平台逐个检索**

	先确定竞品清单，再按每个竞品分别检索；不要只用行业泛词反复搜索。

	按目标市场从 `references/platform-search.md` 选择目标平台，并对每个竞品执行“应用商店基线 + 必跑渠道 + 兜底渠道”的顺序。具体平台范围以 `platform-search.md` 为准，不在 `SKILL.md` 中临时决定渠道清单。

	第一步：为每个直接竞品找 1 个且仅 1 个非 iOS 应用商店截图来源：

	- 中国大陆：优先比较小米应用商店、应用宝、华为/OPPO/vivo、酷安、豌豆荚、APK 镜像等能公开看到截图的 Android 来源。
	- 全球/欧美、东南亚/亚洲：优先 Google Play；如果 Google Play 不可用，再考虑 APKPure、APKCombo、AppBrain 等可公开核验截图的非 iOS 来源。
	- 不调用 iOS App Store，不返回 Apple App Store；同一竞品命中多个非 iOS 应用商店时，只保留截图质量、截图数量、版本/更新时间、市场匹配度和访问稳定性综合最高的一条。

	第二步：按目标市场执行必跑渠道：

	- 中国大陆：UI Notes、Bilibili、Google Images、Google Videos。
	- 全球/欧美：Mobbin、Page Flows、YouTube、Google Images、Google Videos。
	- 东南亚/亚洲：Chamjo、Mobbin、UI Sources、YouTube、Google Images、Google Videos。

	截图/流程库渠道必须优先寻找产品级集合页。UI Notes 命中单张 `/pin/` 截图页时，必须继续检索 `site:uinotes.com/app [竞品名称]`、`[竞品名称] UI Notes app` 或从单张截图页追踪同源 App 集合页；同一产品同时存在 `/app/` 集合页和 `/pin/` 单图时，默认优先返回 `/app/` 集合页，并记录截图数量、版本、功能分类和登录限制。

	第三步：如果必跑渠道召回不足、目标页面覆盖不足或需要交叉核验，再选择 `platform-search.md` 中的其他渠道兜底，例如官网/帮助中心、抖音/TikTok、社区、媒体、APK 镜像、其他截图库或画板。

	每个竞品在每个目标平台中，必须依次检索以下固定查询词：

	- `[竞品名称]`
	- `[竞品名称] APP`
	- `[竞品名称] UI`
	- `[竞品名称] Page`
	- `[竞品名称] 使用教程`
	- `怎么使用[竞品名称]`

	所有查询词都要检索完，不因前几个查询已有结果就停止。检索得到的是候选素材池。

	视频平台不能只依赖上述固定查询词。B站、YouTube、抖音/TikTok 等视频渠道还要按 `platform-search.md` 补充真实用户流程词、录屏词和同合集/同 UP 主追踪，避免漏掉标题不含 `APP` / `UI` 但实际是完整软件录屏的资源。

	如果用户指定功能、页面或状态，优先在候选素材池的下钻核验阶段判断是否覆盖该目标；不要默认把功能词、页面词或状态词拼进每个固定查询词。只有固定查询组召回不足，或目标词本身是强检索词时，才追加目标关键词做补充检索。

	每个进入主结果表的直接竞品，都要至少尝试覆盖 `应用商店`、`界面/截图`、`视频/官方/教程` 中的两类资源。只返回一个应用商店链接时，必须确认对应市场的必跑渠道和必要兜底渠道已检索且没有合格 high fit / medium fit 来源，并在覆盖不足中说明。

4. **下钻核验**

	打开候选页面、视频、截图集合、帮助文档或应用市场截图，确认候选资源是否符合用户检索要求。

	核验内容包括：

	- 素材类型：应用商店截图、截图合集、流程库、视频、帮助文档、官网资料、教程文章、画板/board 等。
	- 可见内容：是否能看到真实 UI、页面、流程、状态、控件、入口或交互动作。
	- 目标匹配：是否覆盖用户指定的竞品、产品类型、市场、功能、页面或状态。
	- 参考价值：界面数量是否足够、信息是否清晰、是否有连续流程、是否比同类结果更有价值。
	- 局限性：是否存在旧版本、地区差异、不同平台、登录限制、视频未完整核验、只展示相邻流程等问题。
	- 元信息：版本、地区、平台、发布时间或来源可信度，能确认时尽量记录。

	不要把搜索结果标题当事实。必须打开来源页、视频页、截图集、应用市场页或帮助文档后再判断。

	Google Images / Google Videos 只作为发现入口，最终必须返回下钻后的来源页，不返回 Google 搜索页、图片直链、缩略图链接、CDN 图片或只打开单张图片的页面。

	花瓣只返回花瓣站内搜索入口或用户提供的花瓣链接，不判断入口内部结果数量、排序或具体画板内容，并标记为 `站内搜索入口 / 需浏览器核验`。

5. **筛选、排序和去重**

	主结果只保留 `high fit` 或 `medium fit`。纯宣传、无可见 UI、无屏幕内容口播、弱相关讨论、无法从页面内容判断相关性的链接不要返回。

	如果单个竞品候选资源超过 6 条，按以下标准优先筛选：

	- 更符合用户指定的功能、页面、状态、市场或平台。
	- 能直接看到真实 UI，而不是只有宣传图或文字描述。
	- 界面数量更多，或能展示连续流程。
	- 来源更可靠，如应用商店、官方帮助中心、成熟截图库、流程库、公开视频。
	- 版本较新，或更接近目标地区/平台。
	- 无需登录、访问稳定、可复核。
	- 与已有结果重复度低，能补充不同页面、流程或状态。

	应用商店截图要单独去重。同一竞品命中多个非 iOS 应用商店时，先比较不同应用商店内的截图数量、清晰度、版本/更新时间、目标市场匹配度和访问稳定性，只返回综合价值最高的一个应用商店链接。即使不同应用商店截图明显不同、覆盖不同平台版本，或用户明确需要多应用商店对比，也不要在主结果表返回多个应用商店链接；多出的只可作为待核验线索或不返回。

	同一产品下必须做去重校验，不返回相同素材链接。相同来源页通过不同查询词命中时，只保留一次。内容高度重复的资源，保留信息量更高、可见 UI 更多或来源更可靠的一条。

	花瓣入口不标 `high fit` 或 `medium fit`，只使用 `站内搜索入口 / 需浏览器核验`、`用户提供 / 需浏览器核验` 或 `自动抓取受限`。

6. **输出结果**

	按 `references/output-format.md` 直接在聊天中输出四列表，不默认生成 HTML 或本地聚合页。

	主结果表按直接竞品聚合，资源分到：

	- `应用商店`
	- `界面/截图`
	- `其他来源`

	如果用户通过编号选择或直接指定了产品，主结果表只包含用户选择/指定的产品，最多 2 个；不要补充其他直接竞品或相邻产品。

	只有在任务明确需要不经用户选择直接汇总行业资源时，主结果表才按原有规则包含 4 个直接相关产品。若候选超过 4 个，按同类程度、目标功能/页面匹配度、应用商店截图可核验度、必跑渠道覆盖度和资源新鲜度筛选前 4 个；其余不要进入主结果表。

	补充参考必须放在独立补充区，不混进主竞品表。用户通过编号选择或直接指定产品时，省略补充参考，不补充非用户指定的资源。补充参考包括：

	- 相邻产品：不是直接竞品，但交互流程相似、公开 UI 素材更丰富的产品，例如同类海外头部产品、相似业务流程产品。
	- 非直接竞品素材：不属于同类产品，但包含相似页面、流程、状态或组件的素材。
	- 行业主题合集：围绕目标行业、功能或页面整理的截图合集、案例合集、灵感合集。
	- App 合集：包含多个相关 App 的榜单、评测、应用合集或截图集合。
	- UI 画板/board：花瓣、Pinterest 或其他画板类资源；按 `references/platform-search.md` 的渠道限制处理。

	每条补充参考都要说明它不是直接竞品，以及可参考的界面、流程或状态。
	未进入用户选择/指定流程时，补充参考必须只输出 2 条资源。优先选择交互相似、素材更丰富的相邻产品；主题合集、App 合集、UI 画板只在更有参考价值时替换进入这 2 条，不为凑数返回低价值链接。

	覆盖不足和待核验线索按 `references/output-format.md` 的格式说明。

## 核验口径

返回链接前至少确认：

- 素材类型：截图、视频、帮助文档、应用市场截图、画板/board、官方页面、教程等。
- 可见内容：页面、状态、控件、入口、流程片段或交互动作。
- 参考价值：为什么对用户目标有用。
- 局限性：旧版本、不同地区、不同平台、只展示相邻流程、只能摘要支撑等。
- Google Images / Google Videos：只作为发现入口；命中结果必须下钻来源页判断，最终返回来源页，不返回搜索页、图片直链、缩略图链接或 CDN 图片。

视频可用性可依据画面、缩略图、章节、字幕、标题或摘要判断；如果未完整播放，必须标注 `画面未完整核验`、`缩略图可见` 或 `仅标题/摘要支撑`，不要编造画面细节。

## 输出前自检

- 是否说明研究范围、检索入口、竞品选择逻辑和素材偏好。
- 是否按用户选择/指定状态正确决定只输出主结果，或输出主结果加补充参考。
- 如果用户未指定具体产品，是否先只输出候选竞品编号清单并等待选择。
- 候选竞品清单是否要求用户最多选择 2 个序号，且没有同时输出资源。
- 如果用户已选择或指定产品，主结果是否只包含用户选择/指定的产品，最多 2 个。
- 用户选择或指定产品后，是否没有补充非用户指定的资源。
- 未进入用户选择/指定流程时，主结果是否只包含 4 个直接相关产品、补充参考是否只包含 2 条资源。
- 是否按目标市场从 `references/platform-search.md` 选择目标平台，并完成对应平台检索。
- 是否为每个直接竞品只保留 1 个非 iOS 应用商店截图来源。
- 是否完全没有调用或返回 iOS App Store / Apple App Store。
- 中国大陆是否已跑 UI Notes、Bilibili、Google Images、Google Videos。
- UI Notes 是否已优先检索 `/app/` 产品集合页；若只命中 `/pin/` 单张截图，是否已继续追踪同源 App 集合页并说明无法确认的原因。
- 全球/欧美是否已跑 Mobbin、Page Flows、YouTube、Google Images、Google Videos。
- 东南亚/亚洲是否已跑 Chamjo、Mobbin、UI Sources、YouTube、Google Images、Google Videos。
- 每个竞品是否在每个目标平台中完成固定查询词检索，且没有因前几个结果可用就提前停止。
- B站/视频平台是否已追加真实流程词、录屏词，并追踪高价值视频的同合集、同 UP 主或相关视频。
- 每个进入主结果表的直接竞品是否至少尝试覆盖两类资源；如果只返回应用商店单链，是否已完成目标平台兜底并写明覆盖不足。
- 如果用户指定功能、页面或状态，是否优先在候选素材池下钻判断，而不是默认把目标词拼进每个固定查询词。
- 写“未找到/覆盖不足”前，是否完成对应渠道和名称变体兜底。
