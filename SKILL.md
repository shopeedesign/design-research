# design-research

用于研究竞品如何处理某个具体产品功能、页面、状态或流程，并收集能直接查看真实 UI 或交互行为的素材：截图、录屏、应用市场截图、截图/流程库、画板/board、帮助中心、教程和带可见界面的官方资料。

不要扩展成泛市场研究、商业策略分析或空泛竞品分析。

## 何时加载 references

- 开始实际检索、选择或核验任何渠道、或准备写“未找到/覆盖不足”时，读取 `references/platform-search.md`。
- 准备最终回答、四列表、fit 标签、覆盖不足或待核验线索时，读取 `references/output-format.md`。
- 只讨论方法、复盘或修改 skill 时，不需要读取平台细则。

## 执行前确认

搜索前先确定：

1. 用户要找的对象：产品类别、目标功能/页面/状态、平台范围和目标市场。
2. 竞品池：用户未指定时，先推断 5-8 个直接竞品；再补 3-5 个交互相似、素材丰富的相邻产品。
3. 名称变体：具体产品要识别别名、旧名/新名、中文名/英文名、包名、开发者名和地区名。
4. 补充关键词：提取用户原始关键词、产品名、页面/功能词、平台词和行业词，供目标平台检索和下钻核验使用。
5. 渠道策略：按 `references/platform-search.md` 选择目标平台；默认优先覆盖 P0，覆盖不足或准备写“未找到”前必须考虑 P1/P2 兜底。


## 核心流程

1. **判断检索入口**

先判断用户给出的检索入口，并选择对应策略：

	- 直接竞品：只要用户指定了具体产品名，就围绕该产品精准搜索；功能、页面、状态或地区词作为下钻关键词，并补充别名、旧名/新名、包名、开发者和地区版本。
	- 产品类型/市场：用户指定了品类、行业或地区市场但未指定具体产品时，先推断相关直接竞品，再逐个按产品名执行信息检索策略；一旦产品进入直接竞品主结果表，就必须按直接竞品最低检索深度处理，不能只返回应用商店单链。
	- 模糊功能：用户只指定功能、页面或状态，且未指定具体产品、品类或市场时，先判断哪些产品或品类可能存在该功能，形成候选竞品池，再按竞品名精准搜索并下钻该功能。
	- 混合输入：如果用户同时给出产品类型/市场和功能，以产品类型/市场为入口，功能词作为下钻筛选；如果同时给出具体产品和功能，以具体产品为入口。
	- 仍无法形成产品、品类、市场或功能关键词时，才向用户追问。

2. **分层选择参考对象**

	结果必须分为两层：

	- 直接竞品：同类产品，作为主结果表的核心对象。
	- 补充参考：无论直接竞品覆盖是否充足，都要另外提供 3-5 个交互相似、素材更丰富的相邻产品，以及非直接竞品素材、行业主题合集、App 合集或 UI 画板。

	补充参考只能放在独立补充区，不能混进主竞品表伪装成直接竞品。先保证直接竞品达到最低覆盖要求，再整理补充参考。

3. **按竞品和目标平台逐个检索**

	先确定竞品清单，再按每个竞品分别检索；不要只用行业泛词反复搜索。

	按目标市场从 `references/platform-search.md` 选择目标平台，并对每个竞品在每个目标平台中执行检索。具体平台范围以 `platform-search.md` 为准，不在 `SKILL.md` 中临时决定渠道清单。

	每个竞品在每个目标平台中，必须依次检索以下固定查询词：

	- `[竞品名称]`
	- `[竞品名称] APP`
	- `[竞品名称] UI`
	- `[竞品名称] Page`
	- `[竞品名称] 使用教程`
	- `怎么使用[竞品名称]`

	所有查询词都要检索完，不因前几个查询已有结果就停止。检索得到的是候选素材池。

	如果用户指定功能、页面或状态，优先在候选素材池的下钻核验阶段判断是否覆盖该目标；不要默认把功能词、页面词或状态词拼进每个固定查询词。只有固定查询组召回不足，或目标词本身是强检索词时，才追加目标关键词做补充检索。

	每个进入主结果表的直接竞品，都要至少尝试覆盖 `应用商店`、`界面/截图`、`视频/官方/教程` 中的两类资源。只返回一个应用商店链接时，必须确认截图/流程库、视频平台、Google Images / Google Videos、官方资料等目标平台已检索且没有合格 high fit / medium fit 来源，并在覆盖不足中说明。

4. **下钻核验**

	打开候选页面、视频、截图集合、帮助文档或应用市场截图，确认候选资源是否符合用户检索要求。

	核验内容包括：

	- 素材类型：应用商店截图、截图合集、流程库、视频、帮助文档、官网资料、教程文章、画板/board 等。
	- 可见内容：是否能看到真实 UI、页面、流程、状态、控件、入口或交互动作。
	- 目标匹配：是否覆盖用户指定的竞品、产品类型、市场、功能、页面或状态。
	- 参考价值：界面数量是否足够、信息是否清晰、是否有连续流程、是否比同类结果更有价值。
	- 局限性：是否存在旧版本、地区差异、不同平台、登录限制、视频未完整核验、只展示相邻流程等问题。
	- 元信息：版本、地区、平台、发布时间或来源可信度，能确认时尽量记录。

	不要把搜索结果标题当事实。必须打开来源页、视频页、截图集、应用市场页或帮助文档后再判断。

	Google Images / Google Videos 只作为发现入口，最终必须返回下钻后的来源页，不返回 Google 搜索页、图片直链、缩略图链接、CDN 图片或只打开单张图片的页面。

	花瓣只返回花瓣站内搜索入口或用户提供的花瓣链接，不判断入口内部结果数量、排序或具体画板内容，并标记为 `站内搜索入口 / 需浏览器核验`。

5. **筛选、排序和去重**

	主结果只保留 `high fit` 或 `medium fit`。纯宣传、无可见 UI、无屏幕内容口播、弱相关讨论、无法从页面内容判断相关性的链接不要返回。

	如果单个竞品候选资源超过 6 条，按以下标准优先筛选：

	- 更符合用户指定的功能、页面、状态、市场或平台。
	- 能直接看到真实 UI，而不是只有宣传图或文字描述。
	- 界面数量更多，或能展示连续流程。
	- 来源更可靠，如应用商店、官方帮助中心、成熟截图库、流程库、公开视频。
	- 版本较新，或更接近目标地区/平台。
	- 无需登录、访问稳定、可复核。
	- 与已有结果重复度低，能补充不同页面、流程或状态。

	应用商店截图要单独去重。同一竞品命中多个应用商店时，先比较不同应用商店内的截图数量、清晰度、版本/更新时间、目标市场匹配度和访问稳定性，只返回综合价值最高的一个应用商店链接。只有不同应用商店截图明显不同、覆盖不同平台版本，或用户明确需要多应用商店对比时，才返回多个应用商店链接。

	同一产品下必须做去重校验，不返回相同素材链接。相同来源页通过不同查询词命中时，只保留一次。内容高度重复的资源，保留信息量更高、可见 UI 更多或来源更可靠的一条。

	花瓣入口不标 `high fit` 或 `medium fit`，只使用 `站内搜索入口 / 需浏览器核验`、`用户提供 / 需浏览器核验` 或 `自动抓取受限`。

6. **输出结果**

	按 `references/output-format.md` 直接在聊天中输出四列表，不默认生成 HTML 或本地聚合页。

	主结果表按直接竞品聚合，资源分到：

	- `应用商店`
	- `界面/截图`
	- `其他来源`

	补充参考必须放在独立补充区，不混进主竞品表。补充参考包括：

	- 相邻产品：不是直接竞品，但交互流程相似、公开 UI 素材更丰富的产品，例如同类海外头部产品、相似业务流程产品。
	- 非直接竞品素材：不属于同类产品，但包含相似页面、流程、状态或组件的素材。
	- 行业主题合集：围绕目标行业、功能或页面整理的截图合集、案例合集、灵感合集。
	- App 合集：包含多个相关 App 的榜单、评测、应用合集或截图集合。
	- UI 画板/board：花瓣、Pinterest 或其他画板类资源；按 `references/platform-search.md` 的渠道限制处理。

	每条补充参考都要说明它不是直接竞品，以及可参考的界面、流程或状态。
	补充参考优先输出 3-5 个相邻产品；主题合集、App 合集、UI 画板各保留最有价值的少量结果，不为凑数返回低价值链接。

	覆盖不足和待核验线索按 `references/output-format.md` 的格式说明。

## 核验口径

返回链接前至少确认：

- 素材类型：截图、视频、帮助文档、应用市场截图、画板/board、官方页面、教程等。
- 可见内容：页面、状态、控件、入口、流程片段或交互动作。
- 参考价值：为什么对用户目标有用。
- 局限性：旧版本、不同地区、不同平台、只展示相邻流程、只能摘要支撑等。
- Google Images / Google Videos：只作为发现入口；命中结果必须下钻来源页判断，最终返回来源页，不返回搜索页、图片直链、缩略图链接或 CDN 图片。

视频可用性可依据画面、缩略图、章节、字幕、标题或摘要判断；如果未完整播放，必须标注 `画面未完整核验`、`缩略图可见` 或 `仅标题/摘要支撑`，不要编造画面细节。

## 输出前自检

- 是否说明研究范围、检索入口、竞品选择逻辑和素材偏好。
- 是否分为直接竞品主结果和补充参考两层。
- 是否按目标市场从 `references/platform-search.md` 选择目标平台，并完成对应平台检索。
- 每个竞品是否在每个目标平台中完成固定查询词检索，且没有因前几个结果可用就提前停止。
- 每个进入主结果表的直接竞品是否至少尝试覆盖两类资源；如果只返回应用商店单链，是否已完成目标平台兜底并写明覆盖不足。
- 如果用户指定功能、页面或状态，是否优先在候选素材池下钻判断，而不是默认把目标词拼进每个固定查询词。
- 写“未找到/覆盖不足”前，是否完成对应渠道和名称变体兜底。
- 主结果是否只保留 `high fit` / `medium fit`。
- 同一竞品命中多个应用商店时，是否默认只返回截图数量、质量和市场匹配度最高的一个应用商店链接。
- 同一产品下是否完成去重，避免重复返回相同素材链接或高度重复来源。
- Google Images / Google Videos 命中的结果是否都下钻到来源页判断，且没有返回搜索页、图片直链或缩略图链接。
- 花瓣是否只返回站内搜索入口或用户提供链接，并标记需浏览器核验。
- 补充参考是否放在独立补充区，并说明不是直接竞品、可参考什么。
