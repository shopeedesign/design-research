# design-research

用于研究竞品如何处理某个具体产品功能、页面、状态或流程，并收集能直接查看真实 UI 或交互行为的素材：截图、录屏、应用市场截图、截图/流程库、画板/board、帮助中心、教程和带可见界面的官方资料。

不要扩展成泛市场研究、商业策略分析或空泛竞品分析。

## 何时加载 references

- 开始实际检索、涉及任何具体渠道、需要花瓣入口、需要 Google Images 补充、或准备写“未找到/覆盖不足”时，读取 `references/platform-search.md`。
- 准备最终回答、四列表、fit 标签、覆盖不足或待核验线索时，读取 `references/output-format.md`。
- 只讨论方法、复盘或修改 skill 时，不需要读取平台细则。

## 执行前确认

搜索前先确定：

1. 用户要找的对象：产品类别、目标功能/页面/状态、平台范围和目标市场。
2. 竞品池：用户未指定时，先推断 5-8 个直接竞品；再补 1-3 个交互相似、素材丰富的相邻产品。
3. 名称变体：具体产品要识别别名、旧名/新名、中文名/英文名、包名、开发者名和地区名。
4. 补充关键词：提取用户原始关键词、产品名、页面/功能词、平台词和行业词，供 Google Images 补充检索使用。
5. 必搜渠道：按 `references/platform-search.md` 选择 P0 渠道；花瓣只返回站内搜索入口；Pinterest 不作为 P0，除非用户明确要求或提供链接。

## 花瓣画板入口规则

花瓣是低自由度渠道。当前环境无法可靠读取花瓣站内搜索结果或画板内容，因此不要筛选、排序或宣称找到具体画板；只返回可点击的花瓣站内搜索入口。

搜索词规则：

- App：`[产品名称] app`
- PC 端/网页端：`[产品名称] UI`

执行要求：

1. 按 `references/platform-search.md` 运行或照抄 `scripts/collection-query-plan.mjs`，生成每个产品的花瓣站内搜索 URL。
2. 不使用 `site:huaban.com/boards`、`site:huaban.com/pins` 或其他外部搜索引擎结果来替代花瓣站内搜索。
3. 不判断花瓣搜索入口里是否有内容、数量多少或前三个画板是什么；状态统一写 `需浏览器核验` 或 `自动抓取受限`。
4. 用户直接提供花瓣画板链接时，保留该链接为 `用户提供 / 需浏览器核验`，但不要自行补充未看见的画板内容。
5. 最终答案保留花瓣入口或用户提供链接；禁止只写“没找到”。

## Pinterest 规则

Pinterest 不作为 P0 渠道，也没有硬闸门。默认不搜索、不返回 Pinterest。只有用户明确要求 Pinterest、提供 Pinterest 链接，或其他 P0/P1 渠道覆盖明显不足且需要灵感补充时，才可作为补充渠道，并标明不是核心竞品证据。

## Google Images 补充规则

Google Images 用于主结果表之外的补充发现，不替代 P0 渠道。用用户关键词、竞品名、页面/功能词和平台词组合检索；先根据图片结果判断是否能看到真实 UI 信息，再打开来源页二次核验。

只返回来源页链接，不返回 Google Images 搜索页、图片 CDN、`imgurl`、`gstatic`、`encrypted-tbn` 或其他图片直链。来源页可以是文章、视频、截图页、应用市场页、教程页或设计案例页，但页面内必须能支撑对应 UI 判断。

## 核心流程

1. **收窄目标**
   把宽泛需求转成可检索的页面或流程，例如首页、商品发布、支付、订单追踪、退款状态、批量导入错误态。只有模糊到无法检索时才追问。

2. **分层选择参考对象**
   先搜同类直接竞品，再搜相邻产品。行业主题合集、App 合集、UI 画板只作为补充参考，不能混进竞品表伪装成竞品。

3. **按竞品和渠道检索**
   每个竞品分别搜索，不要只用泛词反复搜。平台有原生搜索时先用原生搜索，再用外部搜索兜底。视频、应用市场、截图/流程库、画板/board、官方资料要按目标市场覆盖。

4. **固定查询组**
   非视频渠道先搜：`[竞品名称]`、`[竞品名称] APP`、`[竞品名称] UI`、`[竞品名称] Page`。
   视频渠道先搜：`[竞品名称] APP`、`[竞品名称] UI`、`[竞品名称] Page`、`[竞品名称] 使用教程`、`怎么使用[竞品名称]`。
   召回不足或需要精筛时，再加目标功能、同义词、平台词、状态词、页面词和行业主题词。

5. **Google Images 补充检索**
   把关键词放到 Google Images / 图片搜索栏检索。先看图片结果是否展示真实 UI（移动端屏幕、网页界面、流程截图、组件状态、设置页、弹窗或操作路径）；看不出 UI、只有海报/宣传图/Logo/人物/纯插画时直接排除。对疑似有效结果，打开来源页做二次判断，确认页面内仍能看到或说明对应 UI，再进入补充列表候选。

6. **下钻核验**
   打开候选页面、视频、截图集合、帮助文档或应用市场截图，确认素材类型、可见内容、匹配原因、局限性和版本/地区/时间等元信息。不要把搜索结果标题当事实。花瓣入口不做内部结果判断，只说明需要浏览器核验；用户提供的花瓣画板只保留链接和可确认信息。

7. **筛选和归类**
   主结果只放 high fit 或 medium fit。纯宣传、无可见 UI、无屏幕内容口播、弱相关讨论、无法从标题/摘要/缩略图/页面内容判断相关性的链接不要返回。花瓣搜索入口不标 high fit / medium fit；用 `站内搜索入口`、`需浏览器核验` 或 `自动抓取受限` 标记。

8. **输出结果**
   按 `references/output-format.md` 直接在聊天中输出四列表，不默认生成 HTML 或本地聚合页。把资源分到 `应用商店`、`界面/截图`、`其他来源`；在主表之外输出 Google Images 补充列表；并写清覆盖不足和待核验线索。

## 核验口径

返回链接前至少确认：

- 素材类型：截图、视频、帮助文档、应用市场截图、画板/board、官方页面、教程等。
- 可见内容：页面、状态、控件、入口、流程片段或交互动作。
- 参考价值：为什么对用户目标有用。
- 局限性：旧版本、不同地区、不同平台、只展示相邻流程、只能摘要支撑等。
- Google Images 补充：图片结果先能看出 UI 信息，来源页下钻后仍能支撑 UI 判断；最终链接必须是来源页，不是图片直链或 Google 图片搜索页。

视频可用性可依据画面、缩略图、章节、字幕、标题或摘要判断；如果未完整播放，必须标注 `画面未完整核验`、`缩略图可见` 或 `仅标题/摘要支撑`，不要编造画面细节。

## 输出前自检

- 是否说明研究范围、竞品选择逻辑和素材偏好。
- 是否按目标市场覆盖 P0 渠道。
- 花瓣触发时是否返回了正确的站内搜索入口。
- 写“未找到/覆盖不足”前是否完成对应渠道兜底和别名兜底。
- 主结果是否只保留 high/medium fit。
- 是否输出了 Google Images 补充列表，或在没有合格来源页时写明没有符合结果。
- Google Images 补充是否都经过来源页二次判断，且没有返回图片直链。
- 主题合集是否放在独立补充区，而不是混进竞品行。
