# design-research

使用此 skill 研究竞品如何处理某个具体产品功能、模块、页面、状态或交互流程，并收集真实界面素材。

重点关注能够帮助用户查看真实 UI 与交互行为的素材：截图、屏幕录制、使用流程视频、应用市场截图、带可见界面的帮助中心页面、展示真实 UI 的产品页面，以及结构化页面级参考资料。不要把任务扩展成宽泛的市场研究、商业策略分析或泛泛的竞品分析。

## 适用边界

适合处理：

- 找某个功能、页面、状态或流程的竞品 UI 参考。
- 收集截图、屏幕录制、教程视频、用户使用视频、应用市场截图、帮助中心页面或页面级案例。
- 比较不同竞品在同一流程中的信息结构、入口、状态反馈、错误处理或关键交互。
- 为产品设计、交互设计、页面改版、功能补齐或设计评审提供外部参考素材。

不适合处理：

- 泛泛的市场规模、商业模式、增长策略或品牌定位分析。
- 只需要列竞品名称、不需要界面素材的普通竞品调研。
- 没有明确产品类别、功能范围或页面对象的开放式行业研究。

## 能力前提与降级策略

此 skill 最好在具备联网搜索、打开网页、查看图片或视频缩略图/页面内容的环境中使用。

如果当前环境不能联网、不能打开候选页面，或无法查看视频/图片内容：

- 明确说明限制。
- 不要假装已经核验链接或看过界面内容。
- 可改为输出搜索词、候选来源、推荐检索路径和需要用户手动核验的素材清单。
- 只有在标题、摘要、可见页面内容、截图、字幕、缩略图或媒体内容足以支持判断时，才给出 `high fit` 或 `medium fit` 标签。

## 默认处理方式

优先做合理推断并开始研究，不要轻易停下来追问。

当用户没有给出明确竞品，而是给出泛类型功能、页面或交互，例如 `PC 端批量上传`、`后台批量导入`、`表格字段映射`、`批量处理失败状态`，先思考哪些产品、行业后台或工作台天然包含该流程，再把这些具体产品定义为竞品池。不要直接用泛功能词反复搜索。

只有当目标功能或模块过于模糊、无法进行有效搜索时，才向用户提出澄清问题。如果缺少产品名称、市场、地区、平台或偏好的素材类型，应先做合理默认假设，并在最终回答中简短说明。

当用户给出的主题较宽泛时，先将其收窄到最可能有价值的子流程再搜索。好的范围示例包括：

- `配送通知功能`
- `订单追踪页面`
- `联系骑手交互`
- `配送失败流程`
- `订阅付费墙`
- `退款状态页面`
- `结账优惠码`
- `邀请团队成员交互`
- `批量导入错误状态`
- `PC 端批量上传文件`
- `CSV/Excel 导入字段映射`

## 工作流程

1. 识别用户的产品类别、目标功能、平台范围和目标市场。
2. 如果用户没有给出范围，根据用户语言、产品名称、功能场景和平台线索推断市场；无法判断时默认优先查找全球来源，并把中文来源作为补充，在结果开头说明假设。
3. 如果用户没有给出明确竞品，先做“功能存在场景推断”：列出 2-4 类最可能拥有该流程的产品类型，再从每类选出 1-3 个具体产品作为候选竞品。
4. 推断 5-8 个相关竞品，除非用户要求更宽或更窄的范围。
5. 按功能相似度、交互参考价值和素材可获得性确定优先级，而不是只按市场规模排序。
6. 先完成竞品判断，再进入素材搜索。不要在竞品列表不清楚时直接用泛功能词搜索，例如 `最后一公里 app 首页 UI` 或 `PC 批量上传 UI`。
7. 对每个竞品，在每个相关搜索资源中强制执行完整的第一轮固定查询组。视频平台和非视频平台使用不同查询组；每一行都是一次独立搜索，不能组合搜索词，不能因为前一个搜索词已有结果就跳过后面的搜索词。
8. 对每一个独立搜索词得到的结果，都要下钻到结果页面、视频、截图集合、画板、帮助文档或应用市场截图里，匹配用户真正关心的目标页面、状态或流程。只有结果内部能确认看到相关 UI，才允许返回；如果只在搜索标题或摘要里相关，但打开后没有目标界面，不返回。
9. 将目标功能翻译成产品中更可能出现的术语，作为第二轮精筛或补充搜索，而不是第一轮主搜索。例如 `联系骑手` 也可能写作 `contact driver`、`message courier`、`call delivery partner` 或 `rider support`。
10. 在三个来源模块中整理结果：
   - `应用市场/官方来源`：应用宝、华为应用市场、小米应用商店、Google Play、官方网站、帮助中心、APK 或应用聚合页面、版本记录和官方截图。仅当用户明确需要 iOS / iPhone / iPad / App Store 界面素材时使用 App Store。
   - `视频或者直接使用视角`：屏幕录制、使用 walkthrough、教程、带可见 UI 的用户评测、引导视频和真实操作片段。
   - `其他相关截图等`：截图合集、页面画廊、UI 拆解文章、带界面图片的帮助文档、应用镜像页面和页面级参考资料。
11. 按竞品分组返回简洁结果，并说明覆盖不足。

## 搜索策略

搜索素材，而不是搜索泛泛介绍。采用“竞品驱动的矩阵检索”，不要一开始就依赖泛功能词。

### 泛类型需求的竞品推断

当用户没有明确说竞品或产品名称时，先把泛类型功能归因到最常见的软件场景，再检索具体产品。步骤：

1. 判断功能属于哪类用户任务，例如导入数据、上传文件、发布商品、批量建档、批量邀请、批量处理异常。
2. 判断这个任务最常出现在哪些 PC/Web 产品中，例如电商商家后台、CRM、协作表格、项目管理、网盘/云存储、DAM/素材管理、广告投放后台、ERP/WMS、招聘 ATS。
3. 从每类产品中选择具体竞品，并优先选择有公开帮助中心、教程视频、产品截图或素材库页面的产品。
4. 先按资源类型强制跑完第一轮固定查询组，再用功能同义词做第二轮精筛。第一轮固定查询组必须全量执行，不能因为已经找到结果而提前停止。
5. 如果循环搜索多轮后仍没有足够可核验素材，再向用户说明当前素材不足，并请用户缩窄行业、数据类型、平台、用户角色或流程阶段。

例：用户说 `PC 端关于批量上传的交互界面资源`，可以先推断候选竞品池：

- 电商/商家后台：`Shopify`、`Amazon Seller Central`、`淘宝商家后台`、`拼多多商家后台`，关注商品 CSV/Excel 批量导入、图片批量上传、导入结果。
- CRM/营销工具：`HubSpot`、`Salesforce`、`Mailchimp`，关注联系人导入、字段映射、去重、错误校验。
- 协作表格/项目管理：`Airtable`、`monday.com`、`Asana`、`Notion`，关注 CSV 导入、字段类型选择、预览确认。
- 云盘/素材管理：`Google Drive`、`Dropbox`、`Box`、`OneDrive`、`Bynder`，关注拖拽上传、队列、进度、失败重试。

### 第一轮固定查询组

第一轮固定查询组是硬约束。对每个竞品、每个资源平台，必须把对应平台类型下的所有搜索词逐条执行一遍；每一行都是一次独立搜索，不允许组合成一个长查询，不允许因为某个搜索词已经找到结果就跳过剩余搜索词。每次搜索后都要下钻核验，所有搜索词全部跑完后再汇总返回命中结果。

视频平台固定查询组，适用于 `B站`、`YouTube`、`TikTok`、`抖音` 等视频或短视频平台：

- `[竞品名称] APP`
- `[竞品名称] UI`
- `[竞品名称] Page`
- `[竞品名称] 使用教程`
- `怎么使用[竞品名称]`

非视频平台固定查询组，适用于 `花瓣`、`Pinterest`、`Mobbin`、`UI Notes`、`Refero`、`App Fuel`、`Design Vault`、`UI Sources`、应用市场、官方网站、帮助中心、搜索引擎和其他页面/截图来源：

- `[竞品名称]`
- `[竞品名称] APP`
- `[竞品名称] UI`
- `[竞品名称] Page`

第一轮的目的只是发现可用素材入口、画板、截图集合、帮助中心或教程，不要把用户给的精确目标词加入第一轮查询。例如用户说 `顺丰 App 首页列表的花瓣 board`，花瓣是非视频平台，第一轮必须分别搜索 `顺丰`、`顺丰 APP`、`顺丰 UI`、`顺丰 Page`；`首页列表` 只能在打开 board 或第二轮精筛时用来判断是否命中。

第二轮补充查询组，仅在第一轮召回不足或需要精筛目标页面时使用：

- 竞品名称 + 目标功能
- 竞品名称 + 功能同义词
- 竞品名称 + `screenshot`、`screen recording`、`walkthrough`、`tutorial`、`review`、`demo`、`UI`、`app flow`
- 竞品名称 + 平台词，例如 `iOS`、`Android`、`web`、`mobile app`、`desktop`
- 竞品名称 + 状态词，例如 `empty state`、`error state`、`confirmation`、`notification`、`tracking`、`settings`、`checkout`
- 竞品名称 + 页面词，例如 `flow`、`page`、`screen`、`settings`、`dashboard`、`onboarding`、`payment`

不分地区，优先考虑以下界面素材库或产品截图来源：

- `Mobbin`、`UI Notes`、`Refero`、`App Fuel`、`Design Vault`、`UI Sources`

再根据目标市场选择补充来源：

针对中国市场产品，可考虑以下查询词或来源：

- `花瓣`、`B站`、`小红书`、`知乎`、`抖音`、`应用宝`、`华为应用市场`、`小米应用商店`、`酷安`、`豌豆荚`、产品帮助中心、官方产品页面

针对全球产品，可考虑以下查询词或来源：

- `YouTube`、`TikTok`、`Reddit`、`Google Play`、`Product Hunt`、`Pinterest`、`Facebook`、官方帮助中心、产品博客、支持文档。仅当用户明确需要 iOS / iPhone / iPad / App Store 界面素材时，才把 `App Store` 加入搜索来源。

### 资源循环与最低覆盖

按以下顺序选择资源，并对每个竞品重复执行对应平台类型的完整固定查询组。不要因为某个平台、某个搜索词或某个官方应用市场结果已经找到素材就停止后续搜索词：

1. **第一优先级：不分地区的界面素材库或产品截图来源**  
   优先尝试 `Mobbin`、`UI Notes`、`Refero`、`App Fuel`、`Design Vault`、`UI Sources`。在这些非视频平台中必须执行非视频平台固定查询组：`[竞品名称]`、`[竞品名称] APP`、`[竞品名称] UI`、`[竞品名称] Page`。可以用站内搜索、平台搜索页、域名过滤或 `site:` 限定资源范围，但不要把平台名、页面词或功能词合入搜索词本身。
2. **第二优先级：按目标市场补充来源**  
   中国市场产品补查 `花瓣`、`B站`、`应用宝`，再按需要补充 `小红书`、`抖音`、`知乎`、`华为应用市场`、`小米应用商店`、`酷安`、`豌豆荚`、官方帮助中心。  
   全球产品补查 `Google Play`、`YouTube`、`Pinterest`、官方帮助中心，再按需要补充 `TikTok`、`Reddit`、`Product Hunt`、`Facebook`。
   花瓣和 Pinterest 这类 board 型平台需要同时尝试站内搜索或平台搜索页，不要只依赖通用搜索引擎的 `site:` 结果；站内搜索也必须逐条执行非视频平台固定查询组，再进入 board 内核验目标页面。
3. **App Store 特例**  
   默认不要返回 App Store，也不要把 App Store 当作移动端默认来源。只有用户明确提出 `iOS`、`iPhone`、`iPad`、`App Store` 或苹果端界面时，才允许搜索和返回 App Store。
4. **覆盖不足说明**  
   如果第一优先级或目标市场关键来源没有找到可用素材，应在 `覆盖不足` 中说明，例如 `UI Notes 未找到该竞品`、`B站未确认有列表首页操作视频`、`应用宝仅有安装页但截图不足`。

### 搜索结果下钻

搜索结果本身只是候选，不是可返回素材。打开候选后按以下顺序核验：

- 页面或视频是否真的包含可见 UI，而不是纯介绍、口播、下载页或营销页。
- 可见 UI 是否命中用户要的页面、功能、状态或流程；如果只展示相邻页面，可以降为 `medium fit`。
- 截图集合、画板、视频合集要判断内部是否有目标页面线索；如果只能确认集合主题相关但无法逐张核验，放入 `待核验线索`，不要放入主表。
- 如果候选页面没有目标 UI，即使标题包含 `APP`、`UI`、`Page`、`使用教程`，也不要返回。

### 画板与集合型素材

`花瓣`、`Pinterest` 等平台常返回画板、board 或 collection，而不是单张界面截图。处理这类链接时：

- 将其标注为 `画板`、`board` 或 `素材集合`，不要误写成单个界面。
- 如果画板主题明确围绕目标产品、目标功能或同类页面，且缩略图可见相关界面，可以标 `high fit` 或 `medium fit`。
- 如果只能确认画板主题相关，但内部图片需要逐张查看，应说明 `集合相关，但需逐张核验具体页面`。
- 输出示例：`花瓣画板: medium fit，聚合招聘 App/职位列表相关界面素材，适合批量查看列表页和职位卡片，但需逐张核验 [link]`。
- `Pinterest board` 同理：可以作为高相关素材集合来源，但不要把整个 board 当成已核验的单一界面。

搜索前先把功能翻译成产品中更可能出现的术语。例如，`联系骑手` 也可能写作 `contact driver`、`message courier`、`call delivery partner` 或 `rider support`。

如果搜索结果过弱，调整策略：

- 从功能词改搜页面词或状态词。
- 从直接竞品扩展到相邻产品。
- 对泛类型需求，回到“功能存在场景推断”，补充新的产品类别和竞品池后再循环搜索。
- 从中文词扩展到英文词，或反向处理。
- 从官方来源扩展到用户 walkthrough、教程、应用镜像页、截图合集。
- 限定站点搜索，例如 `site:uinotes.com`、`site:huaban.com`、`site:bilibili.com`、`site:youtube.com`、`site:pinterest.com`、`site:support.example.com`、`site:android.myapp.com`、`site:appgallery.huawei.com`、`site:play.google.com`。仅 iOS 明确场景使用 `site:apps.apple.com`。
- 如果多轮搜索仍不足，应再向用户确认是否缩窄到某个行业或任务对象，例如商品、订单、联系人、素材、履约单、SKU、员工、用户、门店或文件。

## 竞品选择

从三类对象中选择竞品：

- 优先选择同一产品类别中的直接竞品。
- 功能高度相似的产品，即使产品类别相邻但不完全相同。
- 对目标流程具有高参考价值的交互模式相似产品。
- 对没有明确竞品的泛类型需求，先选择“最可能拥有该功能的具体产品”，再把它们当作竞品；竞品可以来自不同品类，但必须共享目标流程。

不要追求竞品数量最大化，要追求素材有用性最大化。如果某个知名竞品没有可用的目标功能素材，可以省略，或明确说明没有找到 high fit 素材。

在表格前加入一句简短说明，解释竞品选择逻辑，例如：

`我优先选择了直接竞品，其次补充了在追踪与通知模式上相似的邻近产品。`

## 素材质量

对视频、截图和页面级素材使用严格的 fit 标签：

- `high fit`：清晰展示目标功能、页面、状态或交互。
- `medium fit`：展示了真实 UI 或相关流程，但没有直接展示精确的目标功能。
- `low fit`：弱相关。除非用户明确要求更宽泛的参考，否则不要返回。
- `do not return`：纯宣传、没有可见界面、没有真实使用过程、无法核验、不相关，或弱到无法帮助用户判断。

宁可留空，也不要用低价值链接填充单元格。

不要返回：

- 没有可见 UI 的纯品牌广告。
- 没有屏幕内容的口播视频。
- 没有展示目标流程的泛产品页面。
- 没有截图、视频或具体页面素材的讨论帖。
- 无法从标题、摘要、页面内容、缩略图、字幕或可见媒体判断相关性的链接。

## 素材核验

返回链接前，尽量确认以下信息：

- 素材类型：截图、视频、帮助文档、应用市场截图、画板/board、官方页面、用户教程等。
- 可见内容：具体看到了哪个页面、状态、控件、入口、流程片段或交互动作。
- 匹配原因：它为什么对用户的目标功能有参考价值。
- 局限性：是否只展示相邻流程、旧版本、不同地区、不同平台，或只间接相关。
- 元信息：尽量确认平台、地区、版本、发布时间或访问日期；无法确认时不要猜测。

不要把搜索结果标题当成事实。只有当可见素材支持时，才说“展示了某流程”或“包含某状态”。

## 素材模块

### 应用市场/官方来源

作为轻量背景素材使用。好的来源包括：

- 应用宝
- 华为应用市场
- 小米应用商店
- 酷安
- 豌豆荚
- Google Play
- 官方产品网站
- 官方帮助中心
- 版本记录或更新日志
- 带截图的 APK 或应用聚合页面

默认不要返回 App Store。只有用户明确要求 iOS / iPhone / iPad / App Store 或苹果端界面素材时，才允许把 App Store 作为这一模块的来源。

这一模块应保持简短。用它来确认产品或功能、获取官方截图、识别术语，或寻找版本和历史线索。

### 视频或者直接使用视角

用于具有真实使用价值的素材：

- 用户 walkthrough
- 教程
- 带屏幕录制的产品评测
- 客户或用户操作视频
- 来自引导或帮助内容的可见界面片段

附上 fit 标签和一句相关性理由。示例：

- `high fit: YouTube walkthrough 展示了结账优惠码输入、校验和错误提示 [link]`
- `medium fit: 展示真实订单追踪流程，但没有出现通知入口 [link]`

### 其他相关截图等

用于截图或页面级素材：

- 截图合集
- 画板、board 或 collection
- UI 拆解文章
- 带可见界面图片的帮助文档
- 带页面截图的应用镜像页
- 页面画廊或 App flow 页面
- 带真实产品截图的博客文章

必要时附上 fit 标签和一句相关性理由。

## 输出格式

除非用户要求其他语言，否则用中文返回结果。

开头包含：

- 如果用户没有提供，先用一句话说明你假设的范围、市场、平台和素材偏好。
- 用一句话说明为什么选择这些竞品。

然后返回这个表格：

| Competitor | 应用市场/官方来源 | 视频或者直接使用视角 | 其他相关截图等 |
|---|---|---|---|

规则：

- 每个竞品一行。
- 每个单元格保持简短、易扫读。
- `应用市场/官方来源` 作为背景确认列，保持轻量；只放最有用的官方、应用市场、帮助中心或 APK 截图来源，不追求数量。
- `视频或者直接使用视角` 列默认输出总计 5-8 条结果；优先放能看到真实操作、屏幕录制、教程或 walkthrough 的素材。
- `其他相关截图等` 列默认输出总计 5-8 条结果；优先放截图合集、页面画廊、UI 拆解、画板、帮助文档截图或应用镜像页。
- 后两列的 5-8 条是整张表的目标数量，不是每个竞品每格都放 5-8 条。每个竞品单元格通常 1-2 条即可；如果某个竞品有特别强的素材，可以适当多放。
- 如果严格核验后不足 5 条，不要用 low fit 链接凑数；返回实际可用数量，并在 `覆盖不足` 中说明哪些竞品、搜索资源或素材类型没有找到。
- 只有当多个链接都确实有用时，才在一个单元格中放多个链接。
- 在有帮助时标明来源名称，例如 `应用宝`、`华为应用市场`、`Google Play`、`YouTube`、`B站`、`花瓣画板`、`Pinterest board`、`UI Notes`、`官方帮助中心`。不要默认使用 `App Store`，除非用户明确需要 iOS / iPhone / iPad / App Store 界面素材。
- 对视频、截图和页面级素材使用 fit 标签。
- 每个 `high fit` 或 `medium fit` 链接都应说明可见素材是什么，而不只是贴链接。
- 每个 `high fit` 或 `medium fit` 链接尽量标注平台、地区、年份/发布时间或访问日期；无法确认时可简短说明 `平台未确认` 或 `时间未确认`。
- 没有有价值结果时允许留空。
- 不要强行为每个竞品填满每种素材类型。
- 主表只放已核验或有足够可见线索支撑 fit 标签的素材；不要把未核验候选放进主表。

推荐的单元格模式：

- `来源：fit 标签，看到的素材 + 相关性说明 [link]`

示例：

- `YouTube: high fit，展示实时骑手追踪地图和联系按钮 [link]`
- `官方帮助中心: medium fit，展示通知设置入口，但没有展示配送 push 文案 [link]`
- `应用宝: medium fit，截图包含订单状态页面，可辅助确认移动端信息层级 [link]`
- `花瓣画板: medium fit，聚合招聘 App 列表页和职位卡片素材，需逐张核验具体页面 [link]`

必要时，在表格后增加一个简短的 `覆盖不足` 说明：

- 列出没有找到 high fit 结果的竞品或素材模块。
- 说明 `视频或者直接使用视角` 和 `其他相关截图等` 是否达到各 5-8 条；如果没有达到，说明是严格核验后素材不足，而不是搜索未做。
- 说明结果是否存在地区偏向、时间较旧，或只是间接相关。
- 说明第一优先级素材库和关键市场补充来源中哪些没有找到可用素材。
- 如果用户想继续深入，建议下一步更收窄的搜索方向。

必要时，再增加一个独立的 `待核验线索` 小节：

- 只放可能相关、但当前环境无法打开或无法确认可见内容的候选链接或搜索方向。
- 明确说明为什么未核验，例如无法访问页面、视频内容不可见、只有标题摘要、需要登录或地区受限。
- 给出用户可继续查证的关键词或来源。
- 不要给这些线索标 `high fit` 或 `medium fit`，也不要把它们当作已确认设计素材。

## 质量标准

- 返回更少但更强的链接，而不是冗长嘈杂的列表。
- 对链接进行足够核验，以支撑 fit 标签。
- 不要编造链接、标题、页面内容、截图或视频细节。
- 除非可见内容、字幕、标题、描述或缩略图能支持判断，否则不要声称某个视频展示了某个流程。
- 除非应用市场或官方网站包含有用截图，否则保持这类素材轻量。
- 默认不要返回 App Store；只有用户明确要求 iOS / iPhone / iPad / App Store 或苹果端界面素材时才使用。
- 对花瓣、Pinterest 等画板型结果，明确标注其为素材集合，并说明是否需要逐张核验。
- 让用户能轻松判断哪些链接最值得先打开。
- 当素材不足时，明确说不足，而不是用低价值链接凑满表格。
